# 电商智能客服系统 — 产品需求文档 (PRD)

> 版本：v1.1 | 日期：2026-06-19 | 作者：nobailan  
> 产品形态：Web 应用 + REST API | 目标用户：电商平台客服运营团队

---

## 一、产品概述

### 1.1 产品背景

当前电商客服面临三大痛点：

| 痛点 | 现状 | 影响 |
|------|------|------|
| **响应慢** | 人工客服平均响应 30s+，高峰期排队数分钟 | 用户流失率 15-20% |
| **答案不一致** | 不同客服对同一问题回答不同，法务风险高 | 客诉率 5-8% |
| **成本高** | 中型电商客服团队 20-50 人，年成本百万级 | 利润率压缩 3-5% |

### 1.2 产品定位

**基于知识图谱 + 大语言模型的电商智能客服系统**，以"让用户分不出是 AI 还是真人"为核心体验目标。

### 1.3 目标用户

| 用户角色 | 核心需求 |
|----------|---------|
| **C 端消费者** | 快速获得准确的商品信息、售后政策、下单帮助 |
| **运营团队** | 标准话术管理、问答质量监控、降低人工介入率 |
| **产品/技术团队** | 可量化评估的 AI 效果、低成本的迭代优化路径 |

### 1.4 核心指标（North Star）

| 指标 | v1.0 基线 | v1.1 目标 | v1.1 实际 |
|------|----------|----------|----------|
| **AI 暴露率** | 未测 | < 2% | **0%** |
| 严格正确率（115 题） | ~65% | ≥ 85% | **71.3%** |
| 可用率 | ~80% | ≥ 95% | **99.1%** |
| 平均响应时间 | 4-6s | ≤ 3s | **2.57s** |
| 话术命中时延迟 | N/A | < 500ms | **< 10ms** |
| 售后问题准确率 | ~60% | ≥ 85% | **88%** |

## 三、功能需求

### 3.1 处理链路（三级漏斗）

```
用户问题
  → 缓存查询（LRU，100 条 → 命中 0s 返回）
  → 下单 Agent（TransactionAgent：识别"帮我下单"→ 订单卡片 → 模拟下单）
  → 话术路由（ScriptRouter：20 条模板，关键词 O(1) + Embedding 兜底，命中 < 10ms）
  → 模板匹配（14 个 Cypher 模板，BGE 相似度 > 0.7 命中）
  → LLM 生成 Cypher（8-shot + EXPLAIN 校验 + 重试 ×2）
  → 实体对齐（Neo4j Hybrid Search + difflib 模糊匹配）
  → 执行 Cypher → LLM 生成回答 → 整段返回
```

### 3.2 话术路由模块

- **模板数量**：20 条（退货 2、物流 1、尺码 1、质量 1、客服 1、保修 1、换货 1、发票 1、优惠券 1、投诉 1、送礼推荐 1、手机推荐 1、耳机推荐 1、学生推荐 1、修改地址 1、取消订单 1、颜色尺码 1、订单查询 1）
- **匹配策略**：关键词子串匹配（O(1)）→ Embedding 余弦相似度（阈值 0.85）
- **核心约束**：命中后返回原文，LLM 不参与生成（零 Token、零延迟、零幻觉）

### 3.3 下单 Agent

- **意图识别**：13 个触发词（"帮我下单""我要买""就它了"等）
- **确认机制**：消息 ≤ 6 字 + 精确匹配确认词（"确认""好的""ok"等），防止长句子误触发
- **闭环**：搜索商品 → 订单卡片（Markdown 格式）→ 用户确认 → 模拟下单返回订单号
- **安全**：所有操作为模拟，不产生真实交易

### 3.4 语音输入

- **前端**：MediaRecorder API 录制 webm 音频
- **后端**：Whisper base 模型离线转文字（language=zh）
- **依赖**：ffmpeg（音频解码）

### 3.5 评测体系

- **测试集**：115 条（商品咨询 40、订单查询 27、推荐导购 23、售后问题 25）
- **三模式对比**：纯 LLM / Naive RAG / 完整方案
- **双指标**：严格正确率 + 可用率
- **AI 暴露检测**：自动识别"联系客服""转人工"等自曝词

## 五、技术架构决策

| 决策 | 选择 | 理由 |
|------|------|------|
| 图数据库 | Neo4j | 电商关系天然是图结构，多跳查询 O(1) 每跳，MySQL JOIN N 跳指数级 |
| LLM | DeepSeek-v4-pro | 中文能力强、API 成本比 GPT-4 低 60%、LangChain 生态兼容 |
| Embedding | BGE-small-zh-v1.5 | 中文语义优化、512 维平衡速度与效果、本地部署零延迟 |
| NER | BERT-base-chinese 微调 | 窄领域任务专用小模型优于大模型（快、便宜、可控） |
| 语音识别 | Whisper base | 离线运行，不依赖 Google 服务（避免国内的 network 错误） |
| 前端 | Vanilla JS | 零框架依赖，842 行，加载快，面试展示友好 |
| 话术 vs LLM | 话术优先 | 确定性 > 生成性：高频标准问题不需要 LLM 的"创意" |

## 八、风险与缓解

| 风险 | 影响 | 缓解方案 |
|------|------|---------|
| LLM 生成错误的 Cypher | 返回空结果或错误数据 | EXPLAIN 校验 + 重试 ×2 + Fallback 兜底 |
| 话术覆盖不全 | 长尾问题走 LLM，质量不稳定 | 持续扩充话术模板，监控未命中率 |
| KG 数据不足（20 SPU） | 商品查询准确率偏低（70%） | 扩充 MySQL 源数据，优先补齐品牌中文名映射 |
| 品牌名中英文不匹配 | 用户输"华为"查不到"HUAWEI" | 实体对齐（向量 + 模糊匹配），模板路径也做对齐 |
| 网络依赖 DeepSeek API | API 故障则系统不可用 | 话术层可在无 LLM 时独立工作，后续引入本地模型 |
