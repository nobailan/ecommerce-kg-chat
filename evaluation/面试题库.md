## 一、技术架构（10 题）

### Q1. 为什么选择 Neo4j 而不是在 MySQL 里做 JOIN？

**考察点**：数据库选型决策能力

**参考答案**：电商关系（品牌↔商品↔分类↔属性）是天然图结构。在 MySQL 中做 3 跳以上的 JOIN（如"手机分类下华为品牌的所有 SKU 规格"）需要 4-5 张表 JOIN，性能指数级下降。Neo4j 遍历关系是 O(1) 每跳，且 Cypher 语法天然表达图遍历，比 SQL 多表 JOIN 更直观。另外图数据库方便未来扩展 GraphRAG 等算法。

---

### Q2. 为什么用 DeepSeek 而不是 GPT-4 或 Claude？

**考察点**：模型选型方法论、成本意识

**参考答案**：三维评估：① 中文能力——DeepSeek 对中文理解和生成质量与 GPT-4 持平，且对中文电商术语（如"SPU""SKU""规格"）的理解更好；② 成本——DeepSeek API 价格约为 GPT-4 的 40%，按日均 1 万次查询计算，年节省 10 万+ API 费用；③ 生态——DeepSeek 有 LangChain 官方集成，切换成本低。temperature=0 确保 Cypher 生成的确定性。

---

### Q3. 为什么要单独训练一个 BERT NER 模型，而不是直接用 DeepSeek 做实体抽取？

**考察点**：模型职责分离、成本-性能权衡

**参考答案**：窄领域任务用专用小模型优于通用大模型。BERT NER 推理毫秒级（vs LLM 秒级）、本地免费（vs API 按 token 计费）、确定性输出（vs LLM 可能的幻觉）。这是工程基本原则——不要让大模型做小模型能做的事。BERT 处理的 1000 条商品标题标注数据量也足以微调出可用精度。

---

### Q4. 解释三级漏斗架构的设计逻辑。

**考察点**：系统架构设计能力

**参考答案**：

```
第 1 层：话术路由（20 条模板）
  → 关键词 O(1) 匹配，< 10ms 返回
  → 覆盖 50%+ 高频场景（售后政策、物流、客服）
  → 零 LLM 调用，零 Token 消耗

第 2 层：Cypher 模板匹配（14 个模板）
  → BGE Embedding 相似度 > 0.7 命中
  → 覆盖 80%+ 商品查询（品牌、分类、价格、SKU）
  → 跳过 LLM Cypher 生成，省 3-5s

第 3 层：LLM 兜底
  → 仅长尾复杂问题走此路径
  → 8-shot Prompt + EXPLAIN 校验 + 重试 ×2
  → 3-6s 响应

设计逻辑：每一层过滤一批问题，越往下成本越高、延迟越大。类比 CPU 缓存 L1→L2→L3。
```

### Q5. 实体对齐为什么需要向量检索 + 模糊匹配双重兜底？

**考察点**：RAG 检索策略设计

**参考答案**：向量检索处理语义相似（如"华为"→ embedding 最近的是"HUAWEI"），但纯向量检索有召回上限（k=1 可能漏掉正确结果）。模糊匹配（difflib.SequenceMatcher）做字符串级别的相似度兜底，阈值 0.6。两者互补：向量解决语义差异，模糊匹配解决拼写变体。如果都失败，保留原始值用 CONTAINS 做最终兜底。

---

### Q6. 为什么 Embedding 选 BGE-small-zh-v1.5 而不是用 OpenAI Embedding API？

**考察点**：部署策略、领域适配意识

**参考答案**：BGE 针对中文语义优化（优于多语言模型）、512 维在电商场景精度可接受、本地部署零 API 延迟和费用。OpenAI Embedding API 虽精度略高但增加网络依赖和延迟（每次实体对齐都要调 API）。本地部署还意味着可以离线运行，不依赖外部服务。

---

### Q7. 前端为什么用 Vanilla JS 而不是 React/Vue？

**考察点**：技术选型的务实思维

**参考答案**：这个项目的前端只有 842 行代码，功能是单一的对话界面。引入 React/Vue 意味着增加构建工具链、状态管理、组件库，对于这个规模的项目是过度工程化。Vanilla JS 零依赖、加载快、面试展示友好（面试官一眼能看懂全部代码）。如果是多人协作的大型前端项目，才会考虑框架。

---

### Q8. EXPLAIN 校验 Cypher 能拦截哪些错误？

**考察点**：Cypher/Neo4j 理解深度

**参考答案**：EXPLAIN 在 Neo4j 中是只解析不执行的计划查看命令。它能拦截：① 语法错误（拼写错误、不存在的函数）；② 语义错误（引用了不存在的节点标签或关系类型）；③ 类型错误（对字符串做数值运算）。但它不能拦截逻辑错误——语法正确但查的不是用户想要的（如用户问价格但 Cypher 查了分类）。这需要后面的结果验证层来兜底。

---

### Q9. Whisper 离线语音识别相比浏览器 SpeechRecognition 的优势？

**考察点**：技术方案对比能力

**参考答案**：浏览器 SpeechRecognition（Web Speech API）依赖 Google 服务器，在中国会报 network 错误（Google 服务被墙）。Whisper base 模型本地运行，零网络依赖，且 language="zh" 参数提升中文识别准确率。代价是首次加载需下载 142MB 模型 + CPU 推理 1-2 秒。MediaRecorder 录制 webm → 后端 Whisper 转文字的流程在 Chrome/Edge/Firefox 均可工作，浏览器兼容性更好。

---

### Q10. 这个架构如果用户量从 1 增长到 100 万，瓶颈在哪里？

**考察点**：规模化思维

**参考答案**：第一瓶颈是 DeepSeek API 的并发限制和延迟（外部依赖）。解决方案：① 扩大话术模板覆盖率，让更多查询不经过 LLM；② 引入语义缓存（Embedding 相似度匹配缓存）；③ LLM 调用做请求队列 + 降级策略。第二瓶颈是 Neo4j 单实例，可用读写分离集群。第三瓶颈是 Python GIL，可换用多进程或异步部署。但 80%+ 的优化收益在前两步。

## 二、产品思维（10 题）

### Q11. 这个项目的核心用户价值是什么？用一句话概括。

**考察点**：产品定位能力

**参考答案**：让电商消费者用自然语言获得即时、准确、一致的客服体验，同时让运营方以零边际成本处理 80%+ 的标准化咨询。一句话：**"人机难辨的电商 AI 客服"**。

---

### Q12. 为什么把"AI 暴露率"作为 North Star 指标而不是准确率？

**考察点**：指标设计、用户心理理解

**参考答案**：准确率是技术指标，AI 暴露率是体验指标。从用户角度看：一个回答即使业务上正确，如果末尾来一句"如有问题请联系在线客服"，用户立刻意识到对方是机器人，信任感崩塌。AI 客服的核心竞争力不是"答对多少题"，而是"用户感觉在跟真人聊天"。所以我们把"不能让用户发现对面是 AI"作为最高优先级。

---

### Q13. 如果上线后发现有 5% 的用户反复问同一个问题但得不到满意回答，你会怎么做？

**考察点**：数据驱动的迭代思维

**参考答案**：① 从日志中拉出这 5% 用户的高频问题列表；② 分析失败原因——是话术没覆盖？KG 没数据？LLM 生成错了？③ 如果是话术没覆盖，补充模板（成本最低）；④ 如果是 KG 数据缺失，推动数据团队补齐；⑤ 如果是 LLM 生成不稳定，补充 Few-shot 示例或调整 Prompt。关键是用数据定位到具体问题，而不是猜测。

---

### Q14. 如何评估"对方正在输入..."动画的效果？

**考察点**：A/B 测试设计

**参考答案**：设计 A/B 实验——实验组看到"输入中"动画（1.5s 最小展示时间），对照组直接显示完整回复。核心指标：① 用户满意度评分（对话结束后弹 👍👎）；② 追问率（用户是否继续提问，高追问率 = 回答不令人满意）；③ 对话完成率（用户是否在得到回答后主动结束对话）。我预计实验组的满意度更高，因为"等待感"符合真人对话的心理预期。

---

### Q15. 如果运营团队说"这个话术模板不够用，我们要 200 条"，你怎么回应？

**考察点**：需求管理、优先级判断

**参考答案**：先看数据——当前 20 条模板的覆盖率是多少？如果 20 条已经覆盖了 80% 的高频问题，200 条的边际收益可能很低（长尾问题词频低、维护成本高）。建议：① 先分析未命中日志，找出 Top 10 高频未命中问题，优先补充；② 模板数从 20 扩到 30-40 条即可覆盖绝大多数场景；③ 不要为了数量而数量，每条模板都需要运营审核话术，200 条的维护成本太高。

---

### Q16. 如何说服老板投入资源做这个项目而不是买现成的智能客服 SaaS？

**考察点**：商业论证能力

**参考答案**：三点论证。① 数据主权——自建系统的 KG 数据完全自主可控，SaaS 方案的数据在第三方，切换成本极高；② 定制化——我们的业务有特殊的 SPU/SKU 模型和行业术语，通用 SaaS 的准确率只有 60-70%，自建能做到 88%（售后）；③ 长期 ROI——自建的一次性开发成本 2-3 人月，SaaS 年费 20-50 万且随调用量增长，2 年内自建更便宜。

---

### Q17. 这个项目如果要收费，你会怎么定价？

**考察点**：商业化思维

**参考答案**：按 API 调用量阶梯定价，免费层（月 1000 次）用于获客，Standard（月 1 万-10 万次）¥0.01/次，Enterprise（私有化部署）年费制。核心逻辑：调用量是客户能理解的计费单位，免费层降低试用门槛，Enterprise 锁定大客户。对标阿里店小蜜的按坐席收费模式，我们的按调用量更灵活。

---

### Q18. 如果 DeepSeek API 突然涨价 10 倍，你怎么应对？

**考察点**：风险应对、成本管理

**参考答案**：① 立即扩大话术模板覆盖率（当前 20 条覆盖 50% 问题，扩到 30 条可覆盖 65%+），话术命中不经过 LLM，零成本；② 引入语义缓存，相似问题直接返回缓存结果；③ 评估切换其他模型（如 DeepSeek-v3、通义千问）的迁移成本，LangChain 的抽象层使模型切换只需改一行代码；④ 长期看，当话术+模板覆盖率达到 85%+ 时，LLM 成本对总成本的影响会大幅降低。

---

### Q19. 如何衡量智能客服的 ROI？

**考察点**：业务指标理解

**参考答案**：ROI = （节省的人工客服成本 - 系统运营成本）/ 系统运营成本。节省的成本 = 原需人工客服处理的咨询量 × 人工单次处理成本 × 智能客服解决率。举例：日均 1 万次咨询，人工单次成本 ¥2，智能客服解决 80%，则日节省 ¥1.6 万，年节省 ¥584 万。系统运营成本 = LLM API + Neo4j 服务器 + 运维人力，年约 ¥50 万。ROI ≈ 10 倍。

---

### Q20. 这个项目最大的产品风险是什么？

**考察点**：风险预判

**参考答案**：合规风险——AI 生成的回答如果包含错误信息（如"7 天无理由退货"但实际某品类是 15 天），可能引发消费者权益纠纷。我们的话术路由设计已经部分缓解了这个问题（标准话术由运营审核，LLM 不参与生成），但 LLM 兜底路径的回答仍需要人工抽检或 LLM-as-Judge 自动质检。

## 三、评测与指标（10 题）

### Q21. 为什么评测需要"严格正确率"和"可用率"两个指标？

**考察点**：评测体系设计

**参考答案**：一个指标无法区分"答对了"和"没答对但态度好"。严格正确率衡量系统真正解决问题的能力（给出品牌名、价格、流程步骤等具体信息）。可用率衡量系统不给用户添堵的能力（兜底回复虽然没回答具体问题，但态度友好、有引导价值）。两者结合才能全面评估——严格正确率看"硬实力"，可用率看"软实力"。

---

### Q22. 如何保证评测集的代表性？

**考察点**：测试集设计方法论

**参考答案**：① 四类场景覆盖（商品咨询/订单查询/推荐导购/售后问题），比例参考真实对话分布；② 每类包含"高频简单问题"和"低频复杂问题"；③ 包含"系统应该知道答案"和"系统不可能知道答案"两类（后者验证兜底能力）；④ 包含多字少字、口语化表达等真实用户输入变体。当前 115 条测试集是起点，后续需要根据真实用户日志持续扩充。

---

### Q23. 三模式对比评测为什么重要？

**考察点**：消融实验思维

**参考答案**：三模式对比本质上是消融实验（Ablation Study）。纯 LLM 是基线，Naive RAG（+模板匹配）看第一步优化效果，完整方案（+话术路由）看第二步优化效果。每个模式的提升量都可以量化归因到具体优化手段。这避免了"准确率 71.3% 是好是坏？"的空洞讨论，而是能说出"话术路由贡献了 11% 的准确率提升"。

---

### Q24. auto_label 为什么不能替代人工标注？

**考察点**：评测局限性的认知

**参考答案**：auto_label 基于关键词匹配（如检测回答是否含"品牌""价格""元"等），粒度太粗。它无法区分"回答了品牌名但完全答非所问"和"正确回答了品牌信息"。比如问"华为手机有哪些"，回答列举了"苹果、三星、小米"——auto_label 会判"正确"（因为含品牌名），但人工会判"错误"（没回答华为）。真正的金标评测需要人工逐条标注 + LLM-as-Judge 双重验证。

---

### Q25. AI 暴露检测具体检测什么？

**考察点**：细节理解

**参考答案**：检测回复中是否包含暴露 AI 身份的词汇——"联系客服""转人工""客服热线""投诉专线""点击页面右下角"等。这些词暗示用户"我不行，你去找别人"，是 AI 客服最大的破绽。真人客服不会说"请联系客服"——他们就是客服。v1.1 修复后，所有话术模板都不再出现这些词（如"联系客服"→"我帮您处理"），AI 暴露率降至 0%。

---

### Q26. 如果测试集从 115 扩到 1000 条，评测会有什么变化？

**考察点**：评测集的规模思考

**参考答案**：① 覆盖更多边界 case，尾部问题会暴露更多 LLM 生成的失败案例，整体准确率可能小幅下降（70% → 65% 区间）；② 话术路由的优势更明显——高频问题的占比会下降，但话术命中的绝对数量会增加；③ 评测时间从 15 分钟增加到 2 小时+，需要考虑优化（并行请求、缓存评测结果）；④ 更重要的是代表性而非绝对数量——1000 条但如果 80% 是简单问题，评测结果会虚高。

---

### Q27. 怎么确保评测结果的可复现性？

**考察点**：工程严谨性

**参考答案**：① 测试集版本化管理（test_set.json 入 git）；② API 评测时禁用缓存（?nocache=true），确保每次都是真实请求；③ LLM temperature=0 确保确定性输出；④ 评测脚本记录时间戳、API 版本、环境信息；⑤ 结果以 JSON 格式保存，含每条测试的完整请求/响应。但 DeepSeek API 的服务器端可能有微小的非确定性，这是外部依赖的固有限制。

---

### Q28. 你会用什么指标衡量 LLM 生成 Cypher 的质量？

**考察点**：LLM 输出评估

**参考答案**：三个层次——① 语法正确率（EXPLAIN 通过率），当前 > 95%；② 执行成功率（Cypher 执行不报错），当前 > 90%；③ 语义正确率（返回结果与用户意图匹配），这个最难评估，当前通过人工抽检。长期需要建立标注数据集（50-100 条问题 + 标准 Cypher），计算 EXACT MATCH 和 EXECUTION MATCH（执行结果是否一致）。

---

### Q29. 评测结果显示完整方案比 Naive RAG 只高 2%，这值得做吗？

**考察点**：数据解读能力

**参考答案**：严格正确率只差 2%，但要看其他维度——① AI 暴露率：完整方案 0% vs Naive RAG 1.7%，这是质的差异；② 售后准确率：完整方案 88% vs Naive RAG 87%，差距不大但完整方案的回答是标准话术、可审计、法务合规；③ 响应时间：话术命中时 < 10ms vs Naive RAG 需要走模板或 LLM。2% 的准确率差距不是故事的全部，确定性、一致性、合规性才是话术路由的核心价值。

---

### Q30. 如果老板问"什么时候能做到 95% 准确率"，你怎么回答？

**考察点**：预期管理

**参考答案**：95% 需要两个条件——① KG 数据量起码扩充到 500+ SPU，当前 20 条限制了商品查询的正确率（仅 70%）；② 话术模板从 20 扩到 30-35 条，覆盖更多高频场景。如果这两点做到，乐观估计严格正确率可达 85-90%。但要达到 95%，还需要引入多轮对话（处理模糊问题时主动澄清）、语义缓存、LLM-as-Judge 自动质检等能力。诚实地说，从 71% 到 85% 是数据问题（低成本），从 85% 到 95% 是系统能力问题（需要架构升级）。

## 四、RAG / KG / LLM 深度题（10 题）

### Q31. 知识图谱 + LLM 相比纯 RAG（文档检索 + LLM）有什么优势？

**考察点**：技术方案对比

**参考答案**：① 精确性——KG 的结构化查询（Cypher）返回精确结果，文档检索可能返回"相关但不精确"的片段；② 多跳推理——"华为手机有哪些颜色"需要品牌→商品→属性三跳，KG 可以一次 Cypher 完成，文档检索需要多轮；③ 可解释性——Cypher 查询路径可审计，文档检索的向量相似度是黑盒；④ 数据更新——KG 的增删改是字段级的，文档检索需要重新 Chunk + Embedding。但 KG 的构建成本高于文档检索，适合结构化数据丰富的场景。

---

### Q32. 为什么话术模板要求"命中后 LLM 不参与生成"？

**考察点**：AI 产品设计原则

**参考答案**：三个原因。① 确定性——退货政策、保修条款这类信息必须一字不差，LLM 改写可能引入错误（把"7 天"说成"15 天"）；② 法务合规——运营/法务审核过的标准话术，LLM 改写后合规性无法保证；③ 效率——LLM 生成消耗时间和 Token，直接返回字符串零成本。话术路由本质上是"把 LLM 的创造力用在需要它的地方，不需要的地方用确定性替代"。

---

### Q33. 模板匹配的 Embedding 阈值为什么是 0.7？

**考察点**：参数调优思维

**参考答案**：0.7 是通过观察误匹配和漏匹配样本确定的经验值。阈值太高（0.85）→ 很多应该命中模板的查询漏掉，走了 LLM（浪费时间和 Token）；阈值太低（0.5）→ 不相关的查询也命中模板，返回错误结果。理想方法是：收集 100 条模板相关问题和非相关问题，绘制不同阈值的 Precision-Recall 曲线，选择 F1 最高的点。0.7 在当前场景下是一个合理的平衡点。

---

### Q34. 如果用户说"华为"，对齐到"HUAWEI"，这个映射关系怎么维护？

**考察点**：知识工程实践

**参考答案**：目前靠两层自动对齐：① Neo4j Vector Hybrid Search（向量语义检索 + 全文检索混合）；② difflib 模糊匹配阈值 0.6。长期应该维护一个"实体别名表"（如 {"华为": "HUAWEI", "苹果": "Apple", "三星": "SAMSUNG"}），优先查表再做向量检索。这个别名表可以由运营维护，也可以从用户查询日志中自动挖掘（高频未命中的查询词 → 人工确认后加入别名表）。

---

### Q35. LRU 缓存只做精确匹配（问题完全相同才命中），局限性是什么？

**考察点**：缓存策略设计

**参考答案**："华为有什么手机"和"华为有哪些手机"语义完全相同，但 LRU 精确匹配无法命中。升级方案是语义缓存（Semantic Cache）：对问题做 Embedding，相似度 > 0.95 的问题返回相同缓存结果。代价是① 需要存储问题 Embedding（额外内存）；② 相似度阈值需要调优（太高命中率低，太低可能返回错误缓存）；③ 每次查询需要一次 Embedding 计算（比精确匹配慢，但比 LLM 调用快得多）。

---

### Q36. 8-shot Prompt 中的 8 个示例你是怎么选的？

**考察点**：Prompt Engineering

**参考答案**：按照查询模式维度各选一个代表——品牌查询、分类查询、品牌+分类组合、属性查询、关系查询、SKU 查询、聚合查询、层级导航。目标是让 LLM 看到所有主要的 Cypher 模式。8 个是权衡——从 3 到 8 提升明显（LLM 学会更多模式），从 8 到 20 边际收益递减（同模式只是参数不同）。8 个 ≈ 2000 token，在 DeepSeek 的上下文窗口内占比合理。

---

### Q37. 你觉得 GraphRAG 适合这个项目吗？

**考察点**：前沿技术判断

**参考答案**：适合作为补充但不适合替代。GraphRAG（微软 2024）的核心思想是用 KG 的社区发现来增强 RAG 的检索质量。在我们的场景中，商品的结构化查询（品牌→商品→价格）用纯 KG 查询（Cypher）更直接高效。GraphRAG 的价值在于处理半结构化/非结构化数据——比如商品描述文本、用户评价、使用手册等。理想的架构是：结构化查询走 KG → Cypher，非结构化查询走 GraphRAG。

---

### Q38. 如果 LLM 生成的 Cypher 一直验证失败（3 次都失败），你怎么办？

**考察点**：容错设计

**参考答案**：当前有 5 层 Fallback 链。一层：话术路由命中 → 直接返回。二层：模板匹配命中 → 执行模板 Cypher。三层：LLM 生成 Cypher → EXPLAIN 校验失败 → 注入错误信息重试（×2）。四层：重试全部失败 → 使用预定义的通用 Fallback Cypher（如按关键词选择品牌查询/分类查询/兜底查询）。五层：Fallback Cypher 也失败 → 返回中文兜底消息（"这个问题我需要再查一下..."）。每一层都是上一层的安全网。

---

### Q39. Neo4j 的 APOC 插件在你的项目里有什么用？没装怎么办？

**考察点**：工程细节

**参考答案**：APOC（Awesome Procedures on Cypher）提供了 `apoc.meta.data` 等存储过程，用于自动获取图数据库的 Schema（节点标签、属性、关系类型）。Community Edition 不含 APOC，`refresh_schema()` 调用会失败。我们的应对：捕获异常后自动回退到手动 Schema 构建——直接用 Cypher 查询 `labels()` 和 `type()` 来构建 Schema 字符串。功能完全一致，只是多写了几十行代码。

---

### Q40. 如果让你重新设计这个系统，你会改什么？

**考察点**：反思和改进能力

**参考答案**：两点。① **多轮对话**应该在第一版就做——当前单轮对话无法处理"那华为的呢？""有红色的吗？"这种指代表达，这是用户最自然的交流方式。现在补做需要改前端（Session 管理）、后端（上下文注入）、Prompt（历史消息管理），成本是 v1.0 做的好几倍。② **评测体系**应该先于功能开发——在没有标准测试集的情况下开发功能，就像没有尺子做衣服。如果一开始就有 115 题测试集，每个新功能的效果可以直接量化。

### Q41. 描述一个你在项目中遇到的意外困难，以及怎么解决的。

**考察点**：问题解决能力

**参考答案**：语音输入功能在 Chrome 上报 "network" 错误。查了文档才发现 Chrome 的 Web Speech API 把音频发给 Google 服务器做识别，而 Google 服务在国内被墙。解决方案——用 MediaRecorder API 在前端录音，传到后端用 Whisper base 模型离线识别。意外收获：Whisper 方案支持所有浏览器（不只是 Chrome），且离线运行不依赖外部服务。

### Q44. 你怎么决定 v1.1 的优先级？为什么先做话术路由而不是多轮对话？

**考察点**：优先级判断

**参考答案**：投入产出比（Impact/Effort）。话术路由：Effort 低（写 JSON + 加一个 match 方法，2 天），Impact 高（售后准确率 60%→88%，AI 暴露率 6%→0%）。多轮对话：Effort 高（改前端 Session 管理 + 后端上下文注入 + Prompt 重构 + Token 窗口管理，2 周），Impact 中高（提升模糊查询和追问体验，但基础准确率没解决前做多轮意义有限）。先做低成本高收益的事，再做需要重投入的事。

### Q45. 老板让你下周上线给真实用户用，你怎么办？

**考察点**：风险评估、上线判断

### Q47. 如何看待 AI Agent 的未来发展？

**考察点**：行业视野

**参考答案**：Agent 的核心是"感知→决策→执行→反馈"的闭环。2025-2026 年的趋势是从单 Agent 走向多 Agent 协作（就像我们从单体 LLM 升级到 ScriptRouter + TemplateMatcher + TransactionAgent 的协作）。真正的挑战不在"让 AI 能做事"，而在"让 AI 知道什么时候该做什么事"和"做错了怎么纠正"。我们项目的下单 Agent 就是一个微缩版的 Agent 实践——意图识别→信息检索→生成卡片→确认→执行。

### Q48. 你觉得 AI 客服会取代人工客服吗？

**考察点**：行业判断、辩证思维

**参考答案**：不会完全取代，但会大幅改变人工客服的工作内容。标准化问题（退货政策、物流查询、品牌列表）100% 可自动化。复杂问题（投诉升级、情感安抚、灰色地带协商）仍需人工。未来的客服团队更像"AI 训练师 + 复杂问题专家"——人不再回答重复问题，而是审核 AI 的回答质量、补充 AI 的知识库、处理 AI 升级上来的疑难 case。团队规模可能缩小 60-70%，但剩余岗位的薪资和门槛会更高。

---

### Q49. 你参加过多 Agent Workflow 的开发，这和单 Agent 有什么区别？

**考察点**：多 Agent 实践经验

**参考答案**：关键区别是"任务拆解 + 职责边界"。单 Agent 需要在一个 Prompt 里处理所有逻辑（意图识别→查询→生成），复杂度线性增长，出错率也高。多 Agent 把大任务拆成职责清晰的小 Agent（就像我们的 ScriptRouter、TransactionAgent），每个 Agent 只处理自己擅长的事。优势：① 每个 Agent 更容易优化和测试；② Agent 之间可以并行工作；③ 一个 Agent 出错不影响其他 Agent。挑战：Agent 间的协调和状态传递需要精心设计。

### Q50. 你对这个项目未来 6 个月的发展规划是什么？

**考察点**：产品规划能力

**参考答案**：分三步。第一步（1-2 月）——扩数据：SPU 从 20 到 500+，话术从 20 到 30+，商品咨询准确率从 70% 提升到 80%+。第二步（3-4 月）——加能力：多轮对话记忆、语义缓存、用户反馈机制（👍👎），使系统在"对话自然度"上接近真人。第三步（5-6 月）——做闭环：接入真实订单系统、对接真实物流 API、支持真实的退换货流程。最终目标是让这个系统不仅能回答问题，还能真正帮用户完成购物全流程。
