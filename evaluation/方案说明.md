# 电商智能客服系统 — 三种方案说明

> 撰写日期：2026-06-18  
> 基于 60 条标准测试集的量化评估结果

---

## 一、旧版本用的是什么方案？

旧版本（v1.0）**不是纯 LLM 方案**，而是介于 **Naive RAG** 和 **完整方案** 之间的形态：

- ✅ 已具备：14 个 Cypher 模板匹配、LLM 8-shot Prompt、Cypher 语法校验 + 重试、实体对齐、LRU 缓存、流式 SSE 响应
- ❌ 不具备：话术路由（ScriptRouter）、下单意图 Agent（TransactionAgent）、评估体系

简单来说，v1.0 在本次评测中对应 **Naive RAG（80.0% 准确率）** 的水平，v1.1 在此基础上通过话术路由和下单 Agent 提升到 **83.3%**。

---

## 二、三种方案详解

### 方案 A：纯 LLM 生成（llm_only）

```
用户问题 → LLM 直接生成回答
```

| 维度 | 说明 |
|------|------|
| **工作原理** | 将用户问题和知识图谱 Schema 一起发给 LLM，由 LLM 直接生成 Cypher 查询，执行后返回结果。**不做任何前置优化**。 |
| **优点** | 实现简单，无需维护模板和话术库 |
| **缺点** | LLM 可能生成语法错误的 Cypher；对标准问题也无法保证一致性；每次都要调用 LLM，延迟高（平均 7.5s） |
| **准确率** | **38.3%**（纯靠 LLM 对 Schema 的理解，错误率高） |
| **响应时间** | 7.51s |
| **适用场景** | 仅作为评测基线，不适合生产环境 |

#### 为什么纯 LLM 准确率这么低？

1. **Cypher 生成不稳定**：LLM 有时生成不存在的节点标签或关系类型
2. **实体名称偏差**：用户说"华为"，数据库存的是拼音或英文；LLM 不了解真实数据分布
3. **关联查询困难**：多跳关系（如"华为手机有哪些颜色"）需要多表 JOIN，LLM 容易出错
4. **标准问题也走生成**：像"怎么退货"这种有固定答案的问题，LLM 每次生成的内容都不一样，法务风险高

---

### 方案 B：Naive RAG（no_script）—— 相当于 v1.0

```
用户问题
  → 缓存查询（命中 → 直接返回）
  → 模板匹配（14 个 Cypher 模板，Embedding 相似度 > 0.7 → 命中）
  → LLM Cypher 生成（8-shot Prompt + 校验 + 重试 ×2）
  → 实体对齐（向量检索 + 模糊匹配）
  → Neo4j 执行查询
  → LLM 生成自然语言回答
```

| 维度 | 说明 |
|------|------|
| **核心策略** | **模板匹配优先 + LLM 兜底**。用 Embedding 将用户问题与 14 个预编译 Cypher 模板做相似度匹配，命中则跳过 LLM Cypher 生成环节 |
| **关键优化** | ① 14 个 Cypher 模板覆盖 80%+ 常见商品查询 ② 8-shot Few-shot Prompt 提升 LLM 生成质量 ③ Cypher EXPLAIN 语法校验 + 失败重试（最多 2 次） ④ 实体对齐：Neo4j Vector Hybrid Search + difflib 模糊匹配兜底 ⑤ LRU 缓存：相同问题再次查询直接返回 |
| **准确率** | **80.0%**（相比纯 LLM 提升 **109%**） |
| **响应时间** | 3.10s（相比纯 LLM 减少 **59%**） |
| **适用场景** | 以商品查询为主的电商场景（品牌查询、分类浏览、SKU 规格等） |

#### 为什么 Naive RAG 比纯 LLM 好这么多？

| 优化点 | 带来的提升 |
|--------|-----------|
| 模板匹配 | 80%+ 常见查询跳过 LLM Cypher 生成 → 省 3-5s + 消除生成错误 |
| 8-shot Prompt | LLM 学会了正确的节点标签和关系类型 → 语法错误率大幅降低 |
| EXPLAIN 校验 + 重试 | 拦截错误 Cypher 并自动修正 → 执行成功率从 ~70% 提升到 >90% |
| 实体对齐 | 将"华为"映射为数据库中的真实品牌名 → 查询结果准确率提升 |

#### Naive RAG 的不足（v1.0 的局限）

1. **标准话术问题仍走 LLM**：像"怎么退货""多久发货"这种有固定话术的问题，没有任何模板能做关键词匹配，必须走 LLM 生成
2. **回答一致性无法保证**：同一个问题每次生成的回答措辞不同，无法通过法务审核
3. **每次生成消耗 Token**：即使是标准售后问题，也要消耗 2000+ Token 生成回答
4. **无法处理交易类意图**：用户说"帮我下单"，系统无法识别并引导

### 方案 C：完整方案（full）—— v1.1 当前版本

```
用户问题
  → 缓存查询
  → 下单 Agent（识别下单意图 → 生成订单卡片 → 模拟执行）
  → 话术路由（16 条标准话术，关键词 + Embedding 匹配 > 0.85 → 直接返回原文，禁止 LLM 改写）
  → 模板匹配（14 个 Cypher 模板）
  → LLM Cypher 生成 + 校验 + 重试
  → 实体对齐
  → Neo4j 执行 + 回答生成
```

| 维度 | 说明 |
|------|------|
| **核心策略** | 在 Naive RAG 基础上，新增 **话术路由（ScriptRouter）** 和 **下单 Agent（TransactionAgent）**，构成「话术优先 → 模板加速 → LLM 兜底」三级漏斗 |
| **新增优化** | ① 16 条标准话术模板覆盖退货/物流/尺码/保修/客服/荐购等高频场景 ② 关键词 O(1) 匹配 + Embedding 余弦相似度兜底 ③ 话术命中后直接返回原文字符串，LLM 完全不参与（零 Token 消耗） ④ 下单意图识别 + 订单确认卡片 + 模拟下单闭环 ⑤ 前端"对方正在输入..."动画替代流式响应，消除 AI 生成痕迹 |
| **准确率** | **83.3%**（相比纯 LLM 提升 **117%**，相比 Naive RAG 提升 4.1%） |
| **响应时间** | 2.90s（话术命中时 < 500ms） |
| **适用场景** | 需兼顾商品查询 + 标准化服务 + 交易闭环的电商智能客服 |

#### 话术路由的实际效果

| 场景 | Naive RAG (v1.0) | 完整方案 (v1.1) |
|------|-----------------|-----------------|
| "怎么退货" | LLM 生成 ~2000 Token 的回答 | 关键词命中 → 直接返回 200 字标准话术（< 10ms） |
| "客服电话是多少" | LLM 尝试从 KG 中查（查不到）→ 兜底回复 | 命中话术模板 → 返回真实客服电话 |
| "帮我下单华为P60" | 无法识别交易意图 → 当成普通查询处理 | TransactionAgent → 搜索商品 → 生成订单卡片 |

#### 完整方案对比 Naive RAG 的增量提升

| 指标 | Naive RAG | 完整方案 | 提升 |
|------|-----------|----------|------|
| 售后问题准确率 | 87% | **93%** | +6% |
| 订单查询准确率 | 79% | **86%** | +7% |
| 平均响应时间 | 3.10s | **2.90s** | -6% |
| 话术命中时 Token | 2000+ | **0** | -100% |

## 三、我们的方案定位

```
┌──────────────────────────────────────────────────────┐
│                    三级漏斗架构                       │
├──────────────────────────────────────────────────────┤
│                                                      │
│  第 1 层：话术路由 (ScriptRouter)                     │
│  ├── 覆盖：退货/物流/客服/售后等标准话术（16 条）       │
│  ├── 匹配：关键词 O(1) + Embedding 余弦相似度           │
│  ├── 命中后：直接返回原文，LLM 不参与，零 Token         │
│  └── 响应：< 500ms                                    │
│                                                      │
│  第 2 层：Cypher 模板匹配                             │
│  ├── 覆盖：品牌查询/分类浏览/价格/SKU 等（14 个）       │
│  ├── 匹配：BGE Embedding 相似度 > 0.7                 │
│  ├── 命中后：跳过 LLM Cypher 生成，直接执行查询         │
│  └── 响应：1.5-3s                                     │
│                                                      │
│  第 3 层：LLM 生成 + Neo4j 查询                       │
│  ├── 覆盖：复杂/长尾问题                               │
│  ├── 8-shot Prompt + EXPLAIN 校验 + 重试 ×2           │
│  └── 响应：3-6s                                       │
│                                                      │
│  特殊层：下单 Agent (TransactionAgent)                 │
│  └── 识别交易意图 → 订单卡片 → 模拟闭环                 │
│                                                      │
└──────────────────────────────────────────────────────┘
```

我们的方案本质上是 **「混合检索增强 + 结构化路由」**：
- **不是纯生成**（方案 A）：有模板和话术作为确定性兜底
- **不是传统 RAG**（方案 B）：有知识图谱作为结构化数据源，比文档检索更精确
- **是三级漏斗架构**：话术（确定性）→ 模板（半确定性）→ LLM（生成式），每层过滤掉一批问题

## 四、后续优化方向

### 短期（1-2 周）

| 方向 | 预期收益 | 说明 |
|------|----------|------|
| **扩充话术模板** | 准确率 +5~8% | 当前 16 条覆盖约 50% 高流量问题，扩充到 30 条可覆盖 80% |
| **语义缓存** | 响应 -50%（重复查询） | 当前 LRU 只做精确匹配，"华为手机"和"华为有什么手机"语义相同但无法命中；升级为 Embedding 语义缓存（相似度 > 0.95 命中） |
| **测试集标注优化** | 评测更精准 | 当前 auto_label 规则较粗粒度；引入人工标注金标测试集，增加 LLM-as-Judge 自动打分 |
| **优化 8-shot Prompt** | 准确率 +3~5% | 根据失败 case 分析，针对性补强 Few-shot 示例；如增加价格查询、多条件联合过滤的示例 |

### 中期（1-2 月）

| 方向 | 预期收益 | 说明 |
|------|----------|------|
| **多轮对话** | 用户体验大幅提升 | 支持上下文关联提问（"那华为的呢？""有红色的吗？"），需管理 Session 和 Token 窗口 |
| **话术模板 A/B 测试框架** | 持续优化依据 | 对比不同话术措辞的转化率/满意度，数据驱动迭代话术 |
| **意图路由 Agent** | 准确率 +5% | 引入 LangGraph 多 Agent 协作，将当前 if-else 路由升级为 LLM 驱动的意图分类 + 动态路由 |
| **Neo4j 数据扩充** | 商品咨询准确率 +10%+ | 当前仅 20 条 SPU，扩充到 500+ 条后模板匹配和 LLM 查询效果都会显著提升 |

### 长期（3-6 月）

| 方向 | 预期收益 | 说明 |
|------|----------|------|
| **GraphRAG** | 复杂查询准确率 +15%+ | 在现有 KG 基础上融合文档向量检索，实现结构化+非结构化混合检索 |
| **个性化推荐** | 增加推荐场景 | 基于用户浏览/对话历史，在 KG 中做协同过滤推荐 |
| **多模态搜索** | 新能力 | 支持用户拍照上传商品图片，用 CLIP 做跨模态检索 |
| **实时数据同步** | 数据新鲜度 | 从批处理 ETL 升级为 MySQL binlog 增量同步，商品信息实时更新 |

## 五、评估方法说明

评测使用 `evaluation/evaluate.py` 脚本，基于 60 条标准测试集（覆盖商品咨询 20 条、订单查询 14 条、推荐导购 11 条、售后问题 15 条），对三种模式逐条发送 API 请求并自动标注。

运行命令：
```bash
# 基线评测（单模式）
python evaluation/evaluate.py --auto

# 三模式对比
python evaluation/evaluate.py --mode compare
```

评测时通过 `?mode=full/no_script/llm_only&nocache=true` 参数控制模式切换和缓存禁用，确保各模式独立评测、互不干扰。
