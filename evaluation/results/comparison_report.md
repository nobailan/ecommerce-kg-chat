# 三模式对比评估报告

> 评估日期：2026-06-19  
> 测试集：115 条标准用例（商品咨询 40、订单查询 27、推荐导购 23、售后问题 25）  
> 评估工具：`evaluation/evaluate.py --mode compare`  
> 版本：v1.1

---

## 一、评估方法

### 三种模式

| 模式 | 说明 |
|------|------|
| **纯 LLM**（llm_only） | 关闭话术路由 + 模板匹配，所有问题均由 LLM 生成 Cypher → 查询 Neo4j → 生成回答 |
| **Naive RAG**（no_script） | 模板匹配 + LLM 兜底，但不启用话术路由。相当于 v1.0 版本 |
| **完整方案**（full） | 话术路由（20 条模板）→ 模板匹配（14 个 Cypher）→ LLM 兜底 + 下单 Agent。即 v1.1 当前版本 |

### 两档指标

| 指标 | 含义 |
|------|------|
| **严格正确率** | 回答包含具体信息（品牌名、价格、流程步骤等），真正解答了用户问题 |
| **可用率** | 严格正确 + 合理兜底引导（虽未给出具体信息但回复得体、有引导价值） |

### AI 暴露检测

回复中出现 `"联系客服"` `"转人工"` `"客服热线"` 等自曝 AI 身份词汇的，**直接判为错误**。真人客服不会说"请联系客服"。

---

## 二、总体对比

| 模式 | 严格正确率 | 可用率 | 响应时间 | 平均 Token | AI 暴露率 |
|------|----------|--------|---------|-----------|----------|
| 纯 LLM | 59.1% | 93.9% | 6.41s | 89 | **6.1%** |
| Naive RAG | 70.4% | 98.3% | 2.60s | 91 | 1.7% |
| **完整方案** | **71.3%** | **99.1%** | **2.57s** | 92 | **0%** |

```
严格正确率提升: +21%（59.1% → 71.3%）
可用率提升:      +5.2%（93.9% → 99.1%）
响应加速:        2.5x（6.41s → 2.57s）
AI 暴露率:       6.1% → 0%（完全消除）
```

## 四、关键发现

### 1. AI 暴露是纯 LLM 的隐形杀手

纯 LLM 在之前的评测中可用率高达 99.1%，引入 AI 暴露检测后暴跌至 93.9%。LLM 生成的自然语言中约 6% 包含"请联系客服""建议咨询在线客服"等自曝表述——用户一眼就能看出对面不是真人。

完整方案通过话术路由将所有售后回复替换为审核过的标准话术（如"我就是您的专属客服呀""告诉我订单号，我帮您处理"），AI 暴露率降为 **0%**。

### 2. 响应时间是完整方案的结构性优势

- 话术命中：关键词 O(1) 匹配，< 10ms 返回
- 模板命中：跳到 Cypher 执行，省去 LLM 生成 3-5s
- LLM 兜底：仅复杂问题走此路径

最终平均响应 2.57s，是纯 LLM（6.41s）的 40%。

### 3. 商品咨询是下一步的瓶颈

商品咨询严格正确率仅 70%，低于售后（88%）。根因在于 Neo4j 中仅 20 条 SPU 数据、品牌名存在中英文不匹配、部分 SPU 缺少 price/color 等属性。扩充 KG 数据量是提升该类别的最直接手段。

### 4. Naive RAG 与完整方案差距不大但方向不同

Naive RAG（69.6% → 70.4%）和完整方案（66.1% → 71.3%）在本次修复后差距缩窄，但两者的"正确"来源完全不同：
- Naive RAG 依赖模板匹配 + LLM，回答一致性无法保证
- 完整方案的话术层提供了确定性、可审计、法务合规的标准回复

---

## 五、结论

v1.1 版本通过三层优化（话术路由 → 模板匹配 → LLM 兜底），在 115 题评测集上达到：

- **严格正确率 71.3%**，领先纯 LLM 21%
- **可用率 99.1%**，接近零失误
- **AI 暴露率 0%**，用户无法从回复中判断对面是 AI
- **响应时间 2.57s**，是纯 LLM 的 2.5 倍速

后续优化方向：扩充 Neo4j 商品数据量（SPU 20→500+）、引入语义缓存、增加话术模板覆盖至 30+ 条、部署 LLM-as-Judge 自动评估替代人工标注。
